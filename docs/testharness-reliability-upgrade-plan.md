# testharness 可靠性升级 — 范围与执行 spec (X1–X6)

> 来源:曹总 2026-06-20 拍板,经喵吉投递。**第一优先级**——所有 RLL 任务经可靠 testharness 验证+QA 把关、"过了能直接投产",是一切效率的基础;它一硬化,四个产品(margay/studio/军师)的迭代速度同时被抬起来。
> 基线:本仓摸底 step `testharness-actual-state-survey` 已 mutual CONSENSUS(Lisa 逐行 spot-check 源码)。X1–X4 出处即该摸底 §X1-X4;X5/X6 是曹总新增、对应摸底 P1 与 TH5。
> 范围:**X1–X6 六件全做**。X1/X5/X6 是"去黑盒、去橡皮图章"的核心,X2/X3/X4 是判定标准与执行链硬化。**X5/X6 不可降级为可选。**

---

## 曹总核心诉求(原话归纳)
1. 测试**过了不知道为什么过、没过说不清原因**——门禁/计划讨论全是 agent↔agent,用户被排除,偏黑盒。
2. 测试报告/证据**形同虚设**——现在只在结束给一段简述,"Lisa 同意了"就当最权威说明,实际一查很多站不住。**用户检查测试报告与证据不可缺少,需要工具 + 流程约定。**

---

## 六件升级

### X1 — 强制"真功能测试行",堵 cascade 跑空 ★核心
- **问题**:每个 code-bearing slice 默认只挂一条 testRunner-dup 的 unit C-row,被去重滤掉 → cascade 合成 tiers 为空 → **没跑功能测试就判过**。实测 L5(端到端)=0%。
- **做什么**:要求每个 code-bearing slice 至少一条**非 testRunner 的真功能/集成 C-row**(harness 命令);缺 → warn→(可选)soft 阻断。
- **出处**:摸底 §X1(`testharness-actual-state-survey.md:73`)。
- **验收**:造一个只有单测行的 slice → 现状 cascade 跑空判过;升级后被挡,必须补真功能行才放行。

### X2 — 质量信号进证据归档
- **问题**:oracle 弱 / 用例三件套(case/expected/assertion)不全,现仅计划期 warn 一句,不进 EvidenceRecord → 报告里看不到"这条判得很弱"。
- **做什么**:weak-oracle / case-triple-incomplete flag 写进 `run-<round>.json`,报告与索引都带质量信号。
- **出处**:摸底 §X2(`:74`)。与 X6 联动(X2 产出的信号正是 X6 要展示给用户检查的)。

### X3 — 启用 oracle 范本库
- **问题**:`cli-cmd.json` 有 per-tier oracle 模板但 `preset.enabled` 未开 → 计划期从零写、写得弱。
- **做什么**:启用 preset,给计划期提供 per-tier oracle 范本。
- **出处**:摸底 §X3(`:75`)。

### X4 — 端到端 harness 进 cascade 真跑 ★核心
- **问题**:wezterm-test / playwright-test skill 仅手动,未接进自动 cascade。
- **做什么**:harness skill 命令写进 C-row + 确保 binary 就绪(`ralph-lisa doctor`),per-slice 真功能测试在 cascade 里真跑。
- **出处**:摸底 §X4(`:76`)。与 X1 联动(X1 强制有功能行,X4 让它真能跑)。

### X5 — 用户在环:门禁/计划不黑盒,判定可解释 ★核心(曹总新增)
- **问题**:测试门禁与计划的讨论全是 agent↔agent,用户被排除——过了不知道为什么过、没过说不清原因。
- **做什么**:
  1. **计划期**:测试计划(测什么/怎么判/为什么这样判)推给用户、**可参与讨论/质疑/调整**,不是 agent 自定完就跑。
  2. **判定期**:每个 PASS/FAIL 给**可解释依据**(判过=哪条 oracle 命中什么实际输出;判不过=哪条断言对不上、差在哪)。
  3. 复用 user-in-loop 决策升级框架 + TH4 计划卡/报告卡,做到"**可追问为什么**"这一层。
- **出处**:摸底 P1(计划/结果不向用户展示);`user-in-loop-decision-model-design.md`。
- **验收**:任挑一个判过的 slice,用户问"凭什么判过" → 给得出"C3 oracle 期望 X、实际输出 X、命中"可核对依据。

### X6 — 报告与证据规范化 + 用户可查询检查 ★核心(曹总新增)
- **问题**:报告现在只是结束给一段简述,"Lisa 同意"即权威,一查形同虚设。用户检查环节缺失。
- **做什么**:
  1. **规范化**:报告 + 证据(命令/期望/实际输出/断言结果/录屏-截图)落成**结构化、可机读、可追溯**格式,非自由文字简述。
  2. **可查询分析工具**:用户能**抽查/检索/对比**任一 slice 的报告与证据(哪条用例、跑了什么、证据在哪、断言怎么判)。
  3. **流程约定**:用户检查作为**不可缺的一环**写进流程(关键 slice 用户可抽验证据后才算真闭环),非 Lisa 点头即终局。
- **出处**:摸底 TH5(证据归档已存在但仅流式落盘,缺规范结构+查询+用户检查);承接 X2。
- **验收**:用户对任一历史 slice"查证据" → 能调出每条用例的命令/期望/实际/断言判定/录屏,并能判断哪些站得住、哪些虚。

---

## 执行顺序(内部,一件件做)
1. **X1 + X4 先**(功能测试真进 cascade、真跑)——没有它,X2/X5/X6 展示的都是空内容。
2. **X2 + X3 次**(判定标准硬化 + 范本)——让跑起来的测试判得准。
3. **X5 + X6 贯穿**(用户在环 + 报告证据规范化可查)——让判得准的结果用户看得懂、查得了、信得过。X5/X6 承接 X1/X4 的产出,但设计可并行启动,**不可降级为可选**。

> 建议:本升级作为 super-rll 的 slice 序列,每件拆成有依赖序的 sub-slice,逐个过门禁。X1/X4 完成即可显著抬高"过了能投产"的可信度;X5/X6 是从"agent 自己说了算"变成"用户可信赖"的关键。
