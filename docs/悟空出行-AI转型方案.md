# 悟空出行 AI 转型全栈方案

## 基于 Ralph-Lisa Loop + Margay 的企业级 AI 落地路径

---

## 一、AI 转型的核心命题

AI 转型不是"买工具"或"上系统"，核心是**理顺人和 AI 的关系，重构工作流程**，让人和 AI 真正在一起工作。

### 转型三原则

| 原则 | 含义 | 反模式 |
|------|------|--------|
| **人 AI 协同，而非 AI 替代** | AI 做初稿/执行，人做决策/确认 | 让 AI 全自动运行、人事后兜底 |
| **自顶向下推动，自下向上落实** | 管理层定方向/给资源，一线团队选场景/出成果 | 全公司统一推、或纯靠一线自发 |
| **小团队切入，扩充战果** | 找 1 个痛感最强的团队打样，出成果后复制 | 同时铺 5 个部门、哪个都不深 |

### 悟空出行的问题全景

悟空出行提出的问题横跨五个部门，核心诉求可归纳为三层：

| 层级 | 核心问题 | 本质 |
|------|---------|------|
| **开发层** | 复杂系统重构、代码质量、测试覆盖 | AI 如何安全地参与工程交付 |
| **业务层** | 内容运营、数据分析、竞品监控 | AI 如何自动化业务流程 |
| **基础设施层** | 安全合规、数据隔离、运维风控 | AI 在生产环境如何可控运行 |

**关键洞察**：悟空出行已经在用 AI（Claude Code、小红书笔记辅助等），遇到了典型的"前期提效、后期失控"问题。需要的不是更多 AI 工具，而是**结构化的人 AI 协作框架**。

## 二、产研部门：开发工作流重构

### 核心理念：原型即需求

产研工作的正确流程是：

```
产品原型（设计稿）  ←── 需求的真正来源
      ↓
PRD 文档           ←── 辅助理解原型，不是需求本身
      ↓
RLL 双 Agent 循环   ←── 确保 PRD 准确完善，再进入开发
      ↓
代码实现 + 测试     ←── 对抗性 Review 保障质量
```

**AI 在需求阶段的定位**：不是让 AI 写 PRD，而是让 AI 帮助**验证 PRD 是否准确描述了原型**。RLL 的 Lisa 天然承担这个角色——审核 PRD 的逻辑完整性、与原型的一致性。

### Q1: 10 年复杂系统 + AI 重构能做到什么程度？

**AI 能做深的**：
- 代码理解与文档逆向生成（调用链分析、依赖图生成）
- 模块级重构建议（识别高耦合代码、推荐拆分方案）
- 批量迁移类操作（API 签名变更、框架升级适配）

**必须人兜底的**：
- 业务规则确认（"这个 if-else 为什么存在"只有老人知道）
- 架构决策（微服务拆分边界、数据库分库策略）
- 线上兜底逻辑（异常分支的历史原因）

**方法论**：RLL 的 **[RESEARCH] → [PLAN] → [CODE] → [REVIEW]** 流程天然适配——先让 AI 做调研理解现状，人审 PLAN 确认理解无误，再逐模块执行。

### Q2: 全量重构可行吗？如何拆解？

**不可行**。验证过的路径：**旁路系统逐步替换** > 按模块原地重构。

RLL 的 **Step 机制**天然支持拆解：
1. [RESEARCH] 调研现有系统模块边界 → Lisa 审核
2. [PLAN] 按模块制定重构顺序 → 人工确认
3. [CODE] 逐模块重构 → Lisa 逐模块 review
4. [CODE] 旁路系统对比测试 → 自动化回归

### Q3: 除了"喂文档"，如何让 AI 理解复杂业务？

**核心思路：Markdown + Git 的知识工程**

RAG/向量检索只是一种**检索手段**，不是知识管理方案。企业知识体系的重构应该依靠：

| 手段 | 说明 | 工具支撑 |
|------|------|---------|
| **Markdown 知识文件** | 业务规则、架构决策、接口契约全部用 Markdown 承载 | Git 版本管理 |
| **结构化 Context 注入** | CLAUDE.md / AGENTS.md 定义项目规范，AI 启动时自动加载 | RLL Rule 文件系统 |
| **MCP 工具暴露** | AI 直接调用内部 API 查询实时数据，而非读静态文档 | Margay MCP Server |
| **知识沉淀自动化** | 每轮 review 的共识自动归档为 Markdown 文件 | RLL Memory 机制 |
| **Git 版本化知识** | 所有知识变更可追溯、可 diff、可回滚 | Git |

**为什么不以 RAG 为中心**：
- RAG 是"搜索"，不是"理解"——模型需要的是结构化的上下文，不是检索片段
- Markdown + Git 天然支持版本管理、协作编辑、diff 对比
- AI 可以直接读 Markdown 文件，不需要额外的向量化/索引步骤
- RAG 适合做**辅助检索**（从大量文档中快速定位），但不应作为知识体系的骨架

### Q4: Review 分层机制

这正是 **Ralph-Lisa Loop 的核心价值**：

| 层级 | 覆盖 | 说明 |
|------|------|------|
| **第一层：AI 自动 Review（Lisa）** | ~70% | 代码风格、类型安全、测试覆盖、回归检测 |
| **第二层：交叉模型 Review** | +15% | Claude 写 → Codex 审，不同模型盲区互补 |
| **第三层：人工 Review（仅关键路径）** | ~15% | 核心业务逻辑、架构决策、安全敏感代码 |

**关键设计**：Lisa 的 [NEEDS_WORK] 是**阻塞式**的——不通过就不能合并，而非建议式。

### Q5: "前期提效、后期失控"的本质原因

**本质**：缺乏**结构化约束**。AI 生成代码很快，但没有 review、没有测试、没有架构规范，技术债累积速度远超人工开发。

**RLL 的解决机制**：

| 阶段 | 失控原因 | RLL 约束 |
|------|---------|---------|
| 早期 | 无 review 直接合并 | Lisa 强制 review，[NEEDS_WORK] 阻塞 |
| 中期 | 测试缺失 | [CODE] 提交**必须**包含测试结果 |
| 后期 | 架构腐化 | 第一轮强制 [PLAN]，人审架构决策 |
| 持续 | 知识丢失 | 每轮共识沉淀到 Markdown + Git |

### Q6-Q8: 测试策略

**分层投入**：
- 优先级 1（必须）: 核心业务路径的集成测试 → AI 生成 + 人确认
- 优先级 2（推荐）: 关键模块的单元测试 → AI 全自动
- 优先级 3（可选）: 边缘场景 → 按 bug 补测试

**防"伪覆盖"三板斧**（来自 Margay 测试框架）：
1. **关键路径优先** — 先定义 P0 路径（用户痛感最大），再检查是否覆盖
2. **状态矩阵** — 每个功能标记 verified / partial / unverified，不看百分比
3. **变异测试** — 故意改代码看测试能否 catch

**AI 可直接上线的测试**：接口契约测试、回归测试、性能基线测试
**需人确认的测试**：核心业务 E2E、安全测试

### Q9-Q10: 需求文档与知识沉淀

**需求阶段的正确流程**：
```
产品经理画原型 → AI 辅助将原型描述为结构化 PRD（Markdown）
→ Lisa 审核 PRD 逻辑完整性 → 人确认 → 进入开发
```

**分散需求归并**：基于 Markdown + Git 的**增量知识沉淀**：
```
Tower 需求 A（v1.0） → AI 提取逻辑 → feature-xxx.md（Git 管理）
Tower 需求 B（v1.1） → AI diff + 合并 → feature-xxx.md 更新（Git commit）
                                      ↓
                          "当前功能逻辑"始终是 HEAD 版本
                          历史演变通过 git log 追溯
```

### Q16: AI 提效量化评估

**推荐指标体系**：

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| **代码交付速度** | 同类型需求 PR 周期对比 | Git 数据 |
| **缺陷密度** | 每千行代码 bug 数对比 | Bug 系统 |
| **Review 轮次** | 平均多少轮通过 review | RLL 自动统计 |
| **测试覆盖变化** | 关键路径覆盖率前后对比 | 测试框架 |
| **认知负载** | 开发者复杂度主观评分 | 问卷 |

**关键**：不用单一指标，用**指标组合 + A/B 对比**（同类需求，一半用 AI 一半不用）。

## 三、基础设施层：安全合规与运维

### 数据安全体系

数据安全不是一个单点方案，而是**分层防御 + 纵深控制**：

#### 第一层：网络与部署隔离

| 措施 | 说明 | 实施要点 |
|------|------|---------|
| **模型私有部署** | 核心业务数据只走内网模型 | 可选方案：vLLM + Qwen/DeepSeek 本地部署 |
| **网络分区** | AI 推理节点在独立 VLAN，与生产数据库隔离 | 只开放必要端口 |
| **API Gateway** | 所有 AI 对内部系统的访问统一走网关 | 鉴权 + 限流 + 审计日志 |

#### 第二层：数据流控制

```
┌── 生产数据 ──────────────────────────────────┐
│                                              │
│  数据脱敏网关                                  │
│  ┌──────────────────────────────────┐        │
│  │ 规则引擎：                        │        │
│  │ - 手机号/身份证/银行卡 → 掩码       │        │
│  │ - 用户姓名 → 脱敏                  │        │
│  │ - Access Key / Token → 完全过滤    │        │
│  │ - 业务数据 → 按角色权限放行         │        │
│  └──────────────────────────────────┘        │
│         ↓                                    │
│  AI 推理层（只能看到脱敏后的数据）               │
│         ↓                                    │
│  输出审计层                                   │
│  ┌──────────────────────────────────┐        │
│  │ - Prompt injection 检测            │        │
│  │ - 敏感信息输出拦截                  │        │
│  │ - 全量操作日志（who/when/what）     │        │
│  └──────────────────────────────────┘        │
└──────────────────────────────────────────────┘
```

#### 第三层：权限与访问控制

| 维度 | 机制 | 说明 |
|------|------|------|
| **最小权限** | AI 只能访问当前任务所需的数据和 API | 通过 MCP 工具声明限定范围 |
| **工作区隔离** | Margay Permission 系统按工作区持久化权限 | `allow_always` 只在该项目有效 |
| **知识库权限** | Markdown 文件通过 Git 仓库权限管理 | 不同团队不同仓库，天然隔离 |
| **检索权限** | RAG 检索（如需使用）在检索前过滤 | 向量存储时附加 access_level 元数据 |

#### 第四层：合规与审计

- **操作全量记录**：每一次 AI 调用的输入/输出/时间戳/操作人
- **决策可回溯**：RLL 的 `.dual-agent/` 目录完整保存每一轮提交和 review
- **定期合规报告**：自动生成 AI 使用审计报告（调用量、异常率、敏感数据触发次数）

### AI 运维体系

悟空出行当前情况：基础监控全覆盖、告警规模不大。**不建议立即投入全面 AI 运维**，应分阶段引入：

#### 阶段一：AI 辅助诊断（低投入、高收益）

| 场景 | 人 AI 分工 | 工具 |
|------|----------|------|
| **告警降噪** | AI 聚合关联告警，人处理聚合后的事件 | 告警系统 + AI 分析 |
| **根因分析** | AI 读日志/指标推理根因，人确认并处理 | 监控系统 + MCP 对接 |
| **变更影响评估** | AI 分析变更影响面，人决定是否执行 | Git + CMDB + AI |

#### 阶段二：AI 辅助执行（需审批网关）

```
AI 生成操作方案（如：扩容 2 台、重启服务 X）
       ↓
审批网关：展示方案 + 影响面 + 风险评估
       ↓
人工审批（一键批准 / 拒绝 / 修改）
       ↓
自动执行 + 结果验证 + 回滚预案
```

**核心原则**：
- 核心业务永远需要人审批（Level 2）
- 只有经过充分验证的标准化操作才能自动化（如：基于规则的水平扩容）
- 每次自动执行都必须有回滚预案

#### 阶段三：智能运维（远期，需数据积累）

- 故障预测（基于历史模式）
- 容量规划（基于业务趋势）
- 成本优化（基于资源使用分析）

**ROI 拐点**：服务器 > 100 台、日均告警 > 50 条时投入产出比明确。悟空出行目前建议**只做阶段一**。

#### 内网架构改造要点

```
┌── 现有内网 ─────────────────────────────────────┐
│                                                 │
│  新增组件（最小改造）：                             │
│                                                 │
│  1. 数据脱敏代理 → 部署在 AI 节点与数据源之间        │
│  2. AI 推理节点 → 内网 GPU 服务器 / CPU 推理         │
│  3. 审批 API → 对接现有审批流程（钉钉/飞书/自建）     │
│  4. 审计日志 → 对接现有日志系统（ELK/Loki）          │
│                                                 │
│  不需要改造：                                      │
│  - 现有监控体系照常运行                              │
│  - 现有网络架构不变                                 │
│  - AI 节点作为"只读消费者"接入                       │
└─────────────────────────────────────────────────┘
```

### 客服部门

**替代人工（规则明确、重复性高的场景）**：
- 自助服务引导：语音/文字 AI 直接处理固定问答，复杂问题转人工
- 夜间 AI 接待：全 AI 值班 + 工单生成 → 次日人工跟进
- 自动摘要：通话结束后 AI 生成摘要 + 自动填工单（节省 20-30% 后处理）

**增强管理（AI 做分析，人做决策）**：
- 全量质检：AI 自动评分每通电话/聊天 → 人关注异常案例
- 客诉分析：AI 识别高频问题和产品卡点 → 人推动产品改进
- 智能排班：AI 预测来电量 → 人确认排班方案

### 供应商管理部门

| 需求 | 人 AI 协作方式 |
|------|--------------|
| OCR 二次校验 | AI 处理高置信度 case / 低置信度转人工（预计减少 60%+ 人力） |
| 违章扣款审核 | AI 判断关联性 / 异常和边界 case 转人工 |

## 五、整体落地方案

### 三个 Harness

#### Harness 1: 开发测试 Harness（RLL + 测试框架）

```
原型 → PRD（Markdown） → Lisa 审核 → 人确认
                                      ↓
       Ralph (Claude/GPT)  ←→  Lisa (Codex/Gemini)
       写代码 + 写测试     ←→  对抗性 Review
       [PLAN] → [CODE]    ←→  [PASS] / [NEEDS_WORK]
                                      ↓
                          测试金字塔 (L0-L4)
                          L0: 单元测试（AI 全自动）
                          L1: 集成测试（AI 生成 + 人确认）
                          L2: E2E（Playwright 自动化）
                                      ↓
                          知识沉淀（Markdown + Git）
```

```
           业务 Agent 层
  ┌──────┬──────┬──────┬──────┐

  └──┬───┴──┬───┴──┬───┴──┬───┘
     └──────┼──────┘      │
            ↓             │
```

```
         ┌──────────────┐
         │ 定方向：AI 转型战略  │
         │ 给资源：预算 + 人力  │
         │ 设目标：可量化指标   │
         └───────┬──────┘

           试点团队（1 个）
```

### 推荐切入点

| 优先级 | 团队 | 场景 | 原因 |
|--------|------|------|------|
| **P0** | 产研 | RLL 双 Agent 开发 | 痛感最强（代码质量 + 测试），效果最可量化 |
