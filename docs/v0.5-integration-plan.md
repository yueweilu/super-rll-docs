# Ralph-Lisa Loop v0.5 集成方案

**Status**: Lisa reviewed, consensus reached (Round 45-50, 2026-04-18)

---

## 一、目标

让所有主流 IDE 的 vibe coder 能**一步调用 Lisa review,且 Lisa 有跨轮次持久上下文**。

当前 v0.4.1 的核心问题:IDE 用户要切终端跑 5 步 CLI 才能让 Lisa 审码,且 MCP `rll_lisa_review` 是 one-shot 无上下文。

---

## 二、当前能力基线(v0.4.1)

| 功能 | 上下文 | IDE 覆盖 | 用户操作 |
|---|---|---|---|
| `auto --engine` 全自动 | ✅ 持久 | CLI only | 1 条命令 |
| `watch-lisa` 后台 watcher | ✅ 持久 | 需开终端 | 终端常驻 + IDE 跑 5 步 CLI |
| `run-lisa` 单轮 | ❌ one-shot | 需开终端 | 3 步 CLI |
| `review` 一键审 | ❌ one-shot | 任何终端 | 1 条命令 |
| MCP `rll_lisa_review` | ❌ one-shot | 10/11 IDE 可连 | 1 步 MCP(但无上下文) |
| MCP `rll_launch`+`rll_submit` | ✅ 有 session | 10/11 IDE | 2+ 步 MCP |
| 多 IDE init(8 文件) | — | ✅ 全 IDE | `ralph-lisa init` |

**体验断裂点**:
1. IDE vibe coder 切终端 = 违背使用习惯
2. MCP 1 步但无上下文 = Lisa 退化成弱化 GStack
3. MCP 有 server 但 `init` 不自动写 mcp.json = 用户要手配
4. `watch-lisa` 要常驻后台进程 = vibe coder 不想管

---

## 三、核心方案:MCP Hidden Session

### 一句话

> `rll_lisa_review` 是 hidden `external-ralph` session 的透明包装。IDE agent 还是 1 步调用,但 MCP server 内部自动维持持久 Lisa session。

### 架构图

```
IDE agent 调 rll_lisa_review({content: "...", lisa_backend: "codex", auto_approve: true})
  │
  ▼
MCP Server (ralph-lisa mcp-server)
  │
  ├── _sessions Map (by cwd)
  │     │
  │     ├── 有匹配 session? ──────────────── 复用 ✅
  │     │     (lisa_backend + auto_approve 严格相同)
  │     │
  │     ├── 有不匹配 hidden session? ──────── 断开旧 + 创建新 ♻️
  │     │
  │     ├── 有 explicit external-ralph? ───── 检查 config 匹配
  │     │     ├── 匹配 → 复用(inject) ✅
  │     │     └── 不匹配 → 报错 ❌
  │     │
  │     ├── 有 explicit autonomous? ───────── 报错 ❌
  │     │
  │     └── 无 session ────────────────────── 创建 hidden session
  │                                             │
  │                                             ▼
  │                                    TurnCoordinator
  │                                    + Lisa transport(持久连接)
  │                                    + coordinator.injectSubmission()
  │                                             │
  ▼                                             ▼
返回 Lisa 完整 review                    .dual-agent/ 状态更新
```

### Session 行为矩阵(10 种 case,全定义)

| 已有 session | 新请求 | config 检查 | 行为 |
|---|---|---|---|
| 无 | `rll_lisa_review` | — | 创建 hidden session |
| hidden(匹配) | `rll_lisa_review` | backend+approval 相同 | 复用(持久上下文) |
| hidden(不匹配) | `rll_lisa_review` | 任一不同 | 断开旧 + 创建新 hidden |
| hidden | `rll_launch` | — | 断开 hidden + 创建新 explicit |
| explicit(external-ralph,匹配) | `rll_lisa_review` | backend+approval 相同 | 复用(inject) |
| explicit(external-ralph,不匹配) | `rll_lisa_review` | 任一不同 | 报错:"session config mismatch" |
| explicit(autonomous) | `rll_lisa_review` | — | 报错:"autonomous session active" |
| explicit | `rll_launch` | — | 报错:"session exists" |
| explicit(external-ralph) | `rll_submit` | — | 正常 inject |
| explicit(autonomous) | `rll_submit` | — | 报错(现有行为) |

**Config 匹配规则**:
- `lisa_backend` 必须相同(codex ≠ claude,transport 不能混)
- `auto_approve` 必须相同(spawn-time CLI 参数,不可在线切换)
- 两者都相同才算匹配,否则重建

**设计原则**:绝不 silent wrong reuse。参数要么被用(创建/重建),要么被验证(匹配复用),要么报错(不匹配)。

### Session 生命周期

```
创建: lazy,首次 rll_lisa_review 调用时
复用: 后续调用 config 匹配时
重建: config 不匹配时,断开旧 transport + 新建
清理: idle 5 分钟无调用 → 断开 transport + 从 registry 移除
断线: transport 异常 → 从 registry 移除,下次调用 lazy-create
重启: MCP server 进程重启 → registry 丢失(内存态),下次调用透明重建
降级: transport connect 失败 → fallback 到 history.md 摘要注入,标记 [degraded]
```

### 三个入口,一个内核

| 入口 | 目标用户 | 上下文 | 内部路径 |
|---|---|---|---|
| `rll_lisa_review`(1 步 MCP) | IDE vibe coder | ✅ hidden session 持久 | → registry → `injectSubmission` |
| `rll_launch` + `rll_submit`(显式 MCP) | power user | ✅ explicit session | → registry → `injectSubmission` |
| `watch-lisa`(CLI 终端) | Claude Code / 终端用户 | ✅ transport 持久 | → `injectSubmission` |

**全部走同一个 `coordinator.injectSubmission()` 语义**。turn/deadlock/history/policy 不分叉。

**验收标准**:
- 连续 3 次 `rll_lisa_review`,第 2/3 次 Lisa 引用前几次反馈(上下文持久)

- transport 断线 → 透明重建
- idle 5 分钟 → session 自动清理
- 真 AI E2E:MCP server + 真 codex,3 轮 review 有上下文

## 七、风险与缓解

| 风险 | 缓解 |
|---|---|
| hidden session 和 rll_launch 的 registry 冲突 | 统一 Map by cwd,hidden→explicit 断开重建 |
| config mismatch silent wrong reuse | strict matching + 报错(文案锁进测试) |
| codex threadId 在 idle timeout 后失效 | 重建 session → randomUUID 新 threadId |
| IDE MCP config 格式差异 | Phase 2B 逐 IDE 验证,不猜 |
| init 覆盖用户已有 MCP config | marker-aware JSON merge,只操作 "rll" 条目 |
| ralph-lisa 不在 IDE 子进程 PATH | init 用绝对路径;fallback npx |
| MCP server 崩溃 | IDE 重启 MCP 连接,server lazy-create 新 session |
| 文档超前于产品能力 | Phase 3 在 Phase 1+2A **验收通过后**才执行 |
