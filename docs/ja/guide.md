## RLLとは

RLL（Ralph-Lisa Loop）は2エージェントコラボレーションツールです。「AIがコードを書く」ことを2つの独立した役割に分離します：

- **Ralph**（開発者）— 計画、コード、テストを作成
- **Lisa**（レビュアー）— Ralphの作業を独立レビューし、テストの真正性を検証

2つのエージェントがターン制ループで交互に作業し、CLIゲートが審判としてテスト結果、ドキュメント整合性、安全性を自動チェックします。**アーキテクチャ判断は人間が行います。**

## なぜRLLが必要か

単一のAIにコードを書かせる問題点：同じAIが自分の答案を採点することになります。RLLは関心の分離を徹底します：

- AIが一方的に「完了」と宣言したコードを受け取ることはありません
- テスト結果は「通ったように見える」ではなく、Lisaが独立検証します
- すべての提出が記録され追跡可能です

**適した場面**: マルチステップ実装、アーキテクチャ判断、ユーザー向け・セキュリティ関連コード、要件が曖昧な探索。

**過剰**: 単一行修正、十分テスト済みのリファクタリング、個人スクリプト、緊急ホットフィックス。

## クイックスタート

### インストール

Node.js >= 18、Claude Code（Ralph用）、Codex CLI（Lisa用）が必要です。

```bash
npm i -g ralph-lisa-loop
```

検証: `ralph-lisa doctor` で環境チェック。

### 初期化

```bash
cd your-project
ralph-lisa init
```

ロールファイルと `.dual-agent/` セッション状態を作成します。プロジェクトファイルを作成したくない場合は `ralph-lisa init --minimal`。

### タスク開始

```bash
ralph-lisa start "ログイン機能の実装"
```

### または自動実行

```bash
ralph-lisa auto --engine --task "ログイン機能の実装"
```

エンジンモードは両エージェントを自動駆動します。

## コアコンセプト

### ターン制ループ

RalphとLisaが交互に行動します。各ラウンド：一方が提出 → CLIが検証 → もう一方が応答。同時に話すことはありません。

### タグシステム

各提出はタグで始まり、ラウンドタイプを示します：

| Ralphのタグ | 意味 |
|------------|------|
| `[PLAN]` | 第1ラウンド：アーキテクチャと方針 |
| `[CODE]` | 実装（テスト結果必須） |
| `[FIX]` | Lisaのフィードバックに基づく修正 |
| `[CONSENSUS]` | 合意確認、タスク終了 |

| Lisaのタグ | 意味 |
|-----------|------|
| `[PASS]` | レビュー承認 |
| `[NEEDS_WORK]` | 修正必要（理由必須） |

第1ラウンドは必ず `[PLAN]` です——コーディング前にLisaが理解を検証します。

### コンセンサスプロトコル

Lisaの判断は助言的であり、最終決定ではありません。両エージェントが `[CONSENSUS]` を提出して初めてタスクが完了します。

5ラウンドで合意できない場合は `[OVERRIDE]`（意見の相違を記録して続行）または `[HANDOFF]`（人間にエスカレーション）。無限ループはありません。

### 品質ゲート

CLIが提出時に自動検証：テスト結果、ドキュメント整合性、シークレット漏洩、複雑度評価。違反はデフォルトで提出をブロックします。

### タスクタイプ

すべてのタスクが完全なTDDを必要とするわけではありません。fast-pathタイプは不要な儀式をスキップします：

- **code-task**: 完全TDD（デフォルト）
- **doc-task**: ドキュメントのみの変更
- **review-task**: レビュー・レポート作成
- **process-task**: プロトコル・PLANファイル変更

重要なルール: `cli/` のソースコードを変更するなら code-task です。

## 提出要件

`[CODE]` と `[FIX]` の提出には以下が必要：

- **テスト結果**: 実行したコマンド、パス数/失敗数
- 結果は実際の実行によるもの——捏造不可
- 回帰テストと新規テストの両方を報告

LisaはRalphの自己申告を鵜呑みにせず、ゲート結果とテスト実行ログを独立検証します。

## 作業モードの選択

| シナリオ | 推奨 |
|----------|------|
| 日常のIDE開発 | IDEがRalph + 端末で `watch-lisa` |
| 完全自動CLI | `auto --engine --task "..."` |
| クイックレビュー | `review --auto-approve` |
| 手動の細かい制御 | `start` → 手動 `submit-ralph` / `submit-lisa` |

## 環境変数

| 変数 | デフォルト | 目的 |
|------|---------|------|
| `RL_POLICY_MODE` | `block` | ゲートモード: `off` / `warn` / `block` |

完全なCLIリファレンスと環境変数: [CLIリファレンス](reference.html)。

## 重要な注意点

### Git規律

小さなコミット、明確なメッセージ、頻繁にコミット。問題が発生したときの安全ネットは `git reset` です。

### 人間の仲裁者は必須

2つのAIは悪い設計に喜んで合意することがあります。RLLは構造化されたAI支援開発であり、自律開発ではありません。アーキテクチャ判断、セキュリティ境界、ビジネス方針はあなたが決定します。
