# 测试作者指南 — RLL 测试框架 (v0.9.4+)

> 读者对象：工程师 + AI agent（Ralph、Lisa、test-runner 子 agent），负责针对 RLL
> 测试框架编写测试 macro 和 spec。
> 版本基线：cli `ralph-lisa-loop@0.9.4`（§171 + §173 skill 已 ship，§184 子 agent
> 已 ship，§172 fixture 已 dogfood 验证，§185 init-safety 已 ship）。

本指南是**如何编写测试 macro（CLI/终端）和 spec（浏览器）**让测试框架可靠运行的
单一权威来源。它汇总了 §171→§185 dogfood 中踩过的所有坑，让你不必重蹈覆辙。

> ## ⛔ 不要用 `ralph-lisa init` 来做测试
>
> **`ralph-lisa init` 是破坏性的。** 它执行 `fs.rmSync('.dual-agent/',
> {recursive:true, force:true})` — 删掉全部 session 状态：round 历史、2000+ 个
> harness-results 评审文件、token 用量日志、complexity-judge artifact、
> test-execution-log。v0.9.4 起，已有历史时它会**询问 y/N 同意**（交互终端）
> 或在非交互环境 fail-closed 中止；旧版本（≤0.9.3）则无提示直接删。
>
> **做测试根本不需要 init。** skill 命令（`ralph-lisa skill wezterm-test`、
> `skill playwright-test`、`skill list-fixtures`）和 test-runner 子 agent 从全局
> 安装的 binary + 用户级 `~/.claude/agents/test-runner.md` 工作，与任何
> `.dual-agent/` 状态无关。
>
> 只有当你要在某目录开启**全新的双 agent RLL 协作**（Ralph 写 + Lisa 审）时
> 才需要 `ralph-lisa init`。如果该目录有过往 RLL 历史，**先备份**：
> `cp -r .dual-agent .dual-agent.bak`。
>
> 做测试只需在任意位置写一个 fixture JSON 文件，把绝对路径传给 skill 命令。
> 零 init。详见 [工作流配方](#工作流配方)。

## 系统总览

```
你（或 Ralph）
    │
    ├── 1. 写 JSON fixture（CLI 用 macro，浏览器用 spec）
    │
    ├── 2. 选调用方式：
    │       a) 直接 CLI：ralph-lisa skill <wezterm-test|playwright-test> --macro|--spec X.json
    │       b) 经子 agent：Agent({subagent_type: "test-runner", prompt: "..."})
    │
    └── 3. 收到 6 字段报告或 JSON 结果
            ├── RESULT: pass|fail|partial
            ├── STEPS:  N/total green
            ├── TIME:   wall_seconds
            ├── FAIL_REASON: ...
            ├── ARTIFACT: 截图/trace 路径
            └── NEXT: 建议
```

**底层运行的是：**
- §171 wezterm-test-skill → 通过 `wezterm cli` 驱动真实 wezterm pane（共享 mux + pane-id 模式）
- §173 playwright-test-skill → 通过动态加载的 `playwright` 包驱动真实 Chromium
- 两者都遵守 `try/finally` cleanup，pane/浏览器在每条退出路径上都会被清理

## 你有的三个工具

### 工具 1：Macro（`.macro.json`）— 终端/CLI 测试

只有 5 种 step 类型：

| 类型 | 必填字段 | 作用 |
|---|---|---|
| `spawn` | `name`、`cwd?` | 在 cwd 开新 wezterm pane，按 name 注册 |
| `send` | `target`、`text` | 向 pane 输入文本（`\n` 表示回车）|
| `wait-for` | `target`、`text`、`timeout_ms?`（默认 3000）| 每 50ms 轮询 pane 文本直到出现子串或超时 |
| `assert-contains` | `target`、`text` | 一次性子串检查 |
| `kill` | `target` | 关闭 pane（结束时也会自动 kill）|

### 工具 2：Spec（`.spec.json`）— 浏览器/Web 测试

只有 6 种 step 类型：

| 类型 | 必填字段 | 作用 |
|---|---|---|
| `navigate` | `url` | page.goto(url, wait_until=load) |
| `fill` | `selector`、`text` | page.fill(selector, text) |
| `click` | `selector` | page.click(selector) |
| `wait-for-text` | `selector`、`text`、`timeout_ms?`（默认 3000）| 每 50ms 轮询 page.textContent |
| `assert-text` | `selector`、`text` | 对 textContent 做一次性子串检查 |
| `screenshot` | `path` | page.screenshot({path}) — 绝对路径 |

### 工具 3：test-runner 子 agent

当测试输出会污染主 agent 上下文（>100 行），或你想要 intent→fixture 路由时：

```typescript
Agent({
  subagent_type: "test-runner",
  prompt: "验证 margay-cli init 是否正确创建骨架。用 .claude/skill-fixtures/margay-init.macro.json，设 RL_WEZTERM_SKILL_REAL=1。"
})
```

子 agent 约束（机械锁定）：
- 工具：仅 Bash + Read + Grep（不含 Edit/Write — route-only 契约）
- prompt 体明确禁止生成新的 spec/macro
- 永远返回 6 字段报告格式

## Wezterm macro

### 必需的 fixture 结构

```json
{
  "name": "你的-macro-名称",
  "description": "这验证什么、为什么",
  "steps": [
    { "type": "spawn", "name": "agent", "cwd": "/tmp/work-dir" },
    { "type": "send", "target": "agent", "text": "command\n" },
    { "type": "wait-for", "target": "agent", "text": "anchor", "timeout_ms": 30000 },
    { "type": "assert-contains", "target": "agent", "text": "expected" },
    { "type": "kill", "target": "agent" }
  ]
}
```

### 运行

```bash
RL_WEZTERM_SKILL_REAL=1 node cli/dist/cli.js skill wezterm-test --macro path.macro.json --json
```

（或经子 agent：见上文 test-runner 一节。）

### Schema 注意点

- `spawn.cwd` 必须在 macro 运行前就存在。先 `mkdir -p /tmp/work-dir`。
- `send.text` 通过 `wezterm cli send-text --no-paste` 逐字符送入 pane。`\n` 表示回车。
- `wait-for.text` 在 pane scrollback 里按**字面子串**搜索（不是正则）。
- 除 `spawn` 和 `kill` 外，每个 step 都需要 `target` 匹配前面某个 `spawn.name`。

### 编写规则（11 条血泪经验）

#### R1. 集成测试优先真打，别 mock

如果你测的是某个真实 CLI 的行为，就驱动真实 binary。mock 后端会漏掉契约漂移。
§57 我们 mock 了 PostHog，结果真 Cloud 行为不同，花了一周修 mock-only 假设。

#### R2. 用稳定 anchor，别用 banner 文案

banner 文案会随工具版本漂移。❌ 不要：
```json
{ "type": "wait-for", "target": "claude", "text": "How can I help" }
```
Claude Code 2.x 改了 banner，macro 直接失效（见 §172 D1 dogfood retry 1）。

✅ 要：
```json
{ "type": "send", "target": "claude", "text": "claude -p '...' && printf 'D1''-DONE\\n'\n" },
{ "type": "wait-for", "target": "claude", "text": "D1-DONE", "timeout_ms": 600000 }
```
用你自己在工作完成后输出的标记。

#### R3. 标记隔离 — 让 wait-for 子串不出现在输入命令里

关键：`wezterm cli get-text` 返回**整个 pane scrollback，包括你输入的命令行**。
如果 wait-for 子串出现在输入文本里，macro 会在任何工作开始前就毫秒级完成。

❌ 不要：
```json
{ "type": "send", "target": "x", "text": "claude ... && echo DONE\n" },
{ "type": "wait-for", "target": "x", "text": "DONE" }
```
"DONE" 出现在输入行 "echo DONE" 里 → wait-for 立即匹配。

✅ 要 — shell 拼接技巧：
```json
{ "type": "send", "target": "x", "text": "claude ... && printf 'D''ONE\\n'\n" },
{ "type": "wait-for", "target": "x", "text": "DONE" }
```
输入文本里是 `'D''ONE'`（"DONE" 不连续）；只有 printf 的**输出**含 "DONE"。
（见 §172 D1 dogfood retry 2。）

替代方案 — shell 变量间接：
```bash
M=$(printf 'D''ONE'); claude ...; echo "$M"
```

#### R4. 现实的超时

- 真实 CLI agent（claude/codex）build：D1 级任务 30s–10min。默认 `timeout_ms: 3000`
  对 agent 工作**永远是错的**。
- shell 命令：通常 5–30s。
- 网络调用：5–10s。

不确定就设宽松值（900000ms = 15min）。wait-for 每 50ms 轮询，一旦匹配立即退出，
所以上限设高没代价。

#### R5. ANSI 颜色码污染

CLI 会输出颜色码（`\033[32m`），它们出现在 `get-text` 输出里。如果 pane 里是
`\033[32mOK\033[0m`，wait-for 子串 `OK` 可能匹配不到。

解法：
- 启动前设 `NO_COLOR=1`：`NO_COLOR=1 claude ...`
- 或工具支持的话传 `--no-color`
- 或用颜色码破坏不了的更短的独特子串

#### R6. cwd 纪律

`spawn.cwd` 目录必须在 macro 运行前存在。如果你跑 D1 式的从零 build，
在调 macro **之前**先 `mkdir -p /tmp/your-build-dir`。

#### R7. cleanup 自动，但也要显式 kill

每个 spawn 的 pane 在 macro 结束时通过 try/finally 被 kill（§171 cleanup 不变量）。
但显式 `kill` step 是好的文档。顺序：spawn → ... → kill。

#### R8. 一个 pane 对应一个测试关注点

别在一个 pane 里驱动 5 个不同的东西。每个目标系统 spawn 单独的 pane。
macro runner 会顺序处理它们。

#### R9. 别通过内部 mock 来测

macro 里你可能需要 mock 外部服务（如 PostHog Cloud）。在系统边界做（环境变量、
fixture-server URL），**不要** patch 内部模块。mock 内部 = 耦合测试。

#### R10. 测试名 = 行为规格

macro 的 `name` 字段应描述行为，不是实现：
- ✅ `margay-init-creates-skeleton`
- ❌ `margay-init-test`
- ❌ `test-margay-init-call-flow`

#### R11. fixture 内容忠于权威来源

如果你的 macro 测的是某个 spec 文档（如 `docs/§159-3cli-macro-dev-tasks-design.md`）
里描述的行为，把 spec 逐字引到 prompt 里。别意译 — fixture 会偏离原意
（§172 D3 narrow B 抓到过这个）。

## Playwright spec

### 必需的 fixture 结构

```json
{
  "name": "你的-spec-名称",
  "description": "这验证什么 UI 行为",
  "steps": [
    { "type": "navigate", "url": "http://localhost:3001" },
    { "type": "fill", "selector": "[data-testid='input']", "text": "value" },
    { "type": "click", "selector": "[data-testid='submit']" },
    { "type": "wait-for-text", "selector": "[data-testid='result']", "text": "expected", "timeout_ms": 5000 },
    { "type": "assert-text", "selector": "[data-testid='final-state']", "text": "done" },
    { "type": "screenshot", "path": "/tmp/result.png" }
  ]
}
```

### 运行

```bash
RL_PLAYWRIGHT_SKILL_REAL=1 node cli/dist/cli.js skill playwright-test --spec path.spec.json --json
```

如果 cli runtime 没装 playwright：cli 退出码 2 + 安装提示。
要强制测试不可用路径：`RL_PLAYWRIGHT_FORCE_UNAVAILABLE=1`。

### 编写规则

#### P1. Selector 策略（锁定优先级）

永远按此顺序优先：

1. `[data-testid='X']` — 显式测试契约，不受设计改动影响
2. `[role='button']`、`[aria-label='Submit']` — 锚定无障碍属性，稳定性尚可
3. tag + 可见文本经 `text=` — `text=Login` 对按钮有效
4. CSS class 选择器 — ❌ 避免；改版会破坏测试

如果被测系统没有 `data-testid`，去游说开发团队加上。每个元素加一行，一劳永逸。

#### P2. assert-text 前先 wait-for-text

Web 页面有异步加载。永远先 `wait-for-text`（轮询）再 `assert-text`（一次性）：

```json
{ "type": "wait-for-text", "selector": "ul.todos", "text": "buy milk", "timeout_ms": 5000 },
{ "type": "assert-text", "selector": "ul.todos li", "text": "buy milk" }
```

只 `assert-text` 不先 wait，页面没渲染完时会得到 flaky 测试。

#### P3. 现实的有界超时

- 静态页面：1–3s
- API 往返：3–5s
- 慢第三方：8–10s
- 永远别用 30000ms（超时过头阻塞流水线）

#### P4. screenshot 用于诊断，不用于断言

用 `screenshot` 在成功**和**失败时都抓状态供后续检查。别**对**截图做断言
（这里还没有视觉回归）。

```json
{ "type": "screenshot", "path": "/tmp/before-action.png" },
{ "type": "click", "selector": "[data-testid='btn']" },
{ "type": "screenshot", "path": "/tmp/after-action.png" }
```

#### P5. cleanup 自动

浏览器经 `try/finally` 关闭，即使 assert-text 失败也关。进程树清理经有界轮询的
`ps` 计数验证（不留 orphan headless_shell 进程）。

#### P6. URL 纪律

如果端口可配置，别硬编码 `http://localhost:3001`。用环境变量，或在 spec 前面
用单独的 setup macro 做 server 就绪检查。

## 已知问题

### 问题 1：Claude Code 2.x banner 漂移

**现象**：macro 等 "How can I help" 30s 然后超时。
**原因**：Claude Code 2.1.144+ 用了不同的 banner 文案。
**解法**：从交互式 REPL（`claude\n` + 等 banner）切到 print 模式
（`claude -p '<prompt>'` + 你自己的标记）。见 §172 D1 macro retry 1。

### 问题 2：wait-for 标记匹配到输入命令

**现象**：macro 在 <100ms 内完成，尽管需要数分钟实际工作。
**原因**：标记子串出现在 `send.text` 字面量里。
**解法**：shell 拼接技巧（上文 R3）。见 §172 D1 macro retry 2。

### 问题 3：pane 数量随时间增长

**现象**：多次测试运行后留下很多 wezterm pane。
**原因**：macro 在 cleanup 前崩溃，或 `spawn` 成功但后续 step 抛错未走 try/finally。
**解法**：§171 + §182 已实现 try/finally + 模块级 live-handle registry + 信号处理。
如果仍见残留 pane，提一个 §182-followup bug；别手动 `wezterm cli kill-pane` 当 workaround。

### 问题 4：Chromium 子进程不会立即回收

**现象**：spec 结束约 10s 后 `ps -ef | grep chromium` 仍显示 orphan 进程。
**原因**：chromium teardown 是异步的 — `browser.close()` 在进程树回收前就返回。
**解法**：检查 orphan 进程数的测试必须有界轮询（如最多等 10s 让计数回到基线）。
见 §173 R6 narrow。

### 问题 5：macOS pty 池耗尽（§181 R8-R9 根因）

**现象**：`wezterm cli spawn` 报 "Device not configured" 或卡死。
**原因**：macOS `kern.tty.ptmx_max=511` 被泄漏测试运行累积的僵尸 tmux session 占满。
**解法**：§182 在 `cli/src/test-lib/temp-project.ts` ship 了 Layer A（live-handle
registry）+ Layer B（ESRCH-only dead-pid sweep）。如果仍遇到，查
`lsof -nP /dev/ptmx | wc -l` — 健康状态应 <50。

### 问题 6：提交 CODE/FIX 时 §137 log-cite 不匹配

**现象**：`Submission BLOCKED by policy: §149 log-cite mismatch`。
**原因**：Test-Results 声明（`passed=N failed=M total=K`）跟 10min 内的
test-execution-log 条目不匹配。
**解法**：提交前先跑 `ralph-lisa gate`（自动追加新 log 条目），再在提交体里引用
gate 的确切数字。

### 问题 7：`--test-name-pattern` 正则字符类触发 §149 C19

**现象**：§149 C19 漂移检测器报 "pattern not in source file"。
**原因**：§149 C19 用字面子串匹配（`src.includes(pattern)`），不是正则。所以
`§143 C[2389]` 匹配不到 `§143 C2 P3-bump` 这样的 describe 文本。
**解法**：用字面子串 pattern：`§143 C`（匹配所有 §143 describe）或 `§143 C2`
（匹配某一个）。

### 问题 8：`ralph-lisa init` 静默删 `.dual-agent/` 历史（≤0.9.3）

**现象**：在有过往 RLL 使用的目录跑 `ralph-lisa init` 后，所有 round 历史 /
harness-results / 日志都没了。
**原因**：cmdInit（commands.ts:1542）无条件做 `fs.rmSync('.dual-agent/',
{recursive,force})`。≤0.9.3 只打 warning，无确认步骤。
**解法 / 规避**：
- v0.9.4+ 已加守卫：交互终端 y/N 提示；非交互 fail-closed；`--force` 覆盖。
- 任何 ≤0.9.3 版本：永远别用 init 来做测试 — 测试不需要 `.dual-agent/` 状态。
- 任何版本，对有历史的目录做 init 前先 `cp -r .dual-agent .dual-agent.bak`。

## 工作流配方

### 配方 A：测试 CLI agent build 流程（D1 式）

```bash
# 1. 准备
mkdir -p /tmp/feature-build-test

# 2. 写 macro
cat > /tmp/feature-macro.json <<EOF
{
  "name": "feature-build-flow",
  "description": "验证 agent X 按 spec Z 搭出 feature Y",
  "steps": [
    { "type": "spawn", "name": "agent", "cwd": "/tmp/feature-build-test" },
    { "type": "send", "target": "agent", "text": "claude --dangerously-skip-permissions -p 'Build X per spec Z. Files: a.js, b.html, c.sh' && printf 'BUILD''_DONE\\n'\n" },
    { "type": "wait-for", "target": "agent", "text": "BUILD_DONE", "timeout_ms": 900000 },
    { "type": "assert-contains", "target": "agent", "text": "BUILD_DONE" },
    { "type": "kill", "target": "agent" }
  ]
}
EOF

# 3. 运行
RL_WEZTERM_SKILL_REAL=1 ralph-lisa skill wezterm-test --macro /tmp/feature-macro.json --json

# 4. 验证副作用
ls /tmp/feature-build-test/
```

### 配方 B：测试 Web 前端（D1-verify 式）

```bash
# 1. 启动 server（独立进程）
PORT=3001 node server.js &
SERVER_PID=$!
sleep 2

# 2. 写 spec
cat > /tmp/feature-verify.spec.json <<EOF
{
  "name": "feature-frontend-verify",
  "description": "验证 CRUD UI 对 server 工作",
  "steps": [
    { "type": "navigate", "url": "http://localhost:3001" },
    { "type": "fill", "selector": "[data-testid='input']", "text": "test value" },
    { "type": "click", "selector": "[data-testid='submit']" },
    { "type": "wait-for-text", "selector": "[data-testid='result']", "text": "test value", "timeout_ms": 5000 },
    { "type": "assert-text", "selector": "[data-testid='result']", "text": "test value" },
    { "type": "screenshot", "path": "/tmp/feature-result.png" }
  ]
}
EOF

# 3. 运行
RL_PLAYWRIGHT_SKILL_REAL=1 ralph-lisa skill playwright-test --spec /tmp/feature-verify.spec.json --json

# 4. 收尾
kill $SERVER_PID
```

### 配方 C：测试交互式 REPL 流程

```json
{
  "name": "psql-query-flow",
  "steps": [
    { "type": "spawn", "name": "db", "cwd": "/tmp" },
    { "type": "send", "target": "db", "text": "psql -U postgres -d testdb\n" },
    { "type": "wait-for", "target": "db", "text": "testdb=#", "timeout_ms": 5000 },

    { "type": "send", "target": "db", "text": "SELECT COUNT(*) FROM users;\n" },
    { "type": "wait-for", "target": "db", "text": "testdb=#", "timeout_ms": 5000 },
    { "type": "assert-contains", "target": "db", "text": "count" },

    { "type": "send", "target": "db", "text": "\\q\n" },
    { "type": "kill", "target": "db" }
  ]
}
```

关键：每个 send 后都跟一个 wait-for `testdb=#`（psql 提示符）再发下一条。
这是标准 REPL 模式。

### 配方 D：组合 macro（setup）+ spec（verify）

```bash
# 1. macro 搭出制品
```

### 配方 E：开发一个功能 → 自动启动测试框架

目标：你写/开发一个功能，测试框架自动运行 — 没有手动「现在去测它」这一步。
两种机制，看你怎么开发来选。

#### E.1 — 走 RLL 双 agent loop 开发（Ralph 写 / Lisa 审）

用 **§150 smoke-auto-loop**。设 `RL_SMOKE_CMD`，之后每次 Ralph `[CODE]`/`[FIX]`
提交后会自动跑测试框架：

```bash
# 1. TDD-first — 在开发功能之前/同时先写好 fixture
vim /abs/path/feature.spec.json     # 或 .macro.json

# 2. 让 RL_SMOKE_CMD 指向它
export RL_SMOKE_CMD="env RL_PLAYWRIGHT_SKILL_REAL=1 ralph-lisa skill playwright-test --spec /abs/path/feature.spec.json"
#   CLI 功能则用：
#   export RL_SMOKE_CMD="env RL_WEZTERM_SKILL_REAL=1 ralph-lisa skill wezterm-test --macro /abs/path/feature.macro.json"

# 3. 启动 loop
ralph-lisa auto --engine
```

每轮自动行为：
1. Ralph 提交 `[CODE]`/`[FIX]`
2. §150 跑 `RL_SMOKE_CMD`（测试框架）
3. 非零退出 → 失败 JSON 存到 `.dual-agent/smoke-failures/`；下一次 Ralph 提交
   自动把 `Smoke-Failure-Context` 前置进去
4. 连续 3 次 smoke 失败 → `smoke-deadlock.txt` + WeCom `task_failed` 告警
5. 双方 CONSENSUS 时 §70 cascade 重跑完整 PLAN 测试表

opt-out：`RL_SMOKE_AUTO_LOOP_OFF=1`。预算在成功运行或 `ralph-lisa smoke-fail clear` 后重置。

#### E.2 — 纯 Claude Code 开发（不走双 agent loop）

用 Claude Code 的 **PostToolUse hook**。在 `.claude/settings.json`：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "env RL_PLAYWRIGHT_SKILL_REAL=1 ralph-lisa skill playwright-test --spec /abs/path/feature.spec.json --json 2>&1 | tail -15"
      }
    ]
  }
}
```

每次 Claude 编辑或写文件时，hook 触发测试框架并把结果喂回 Claude 上下文。
matcher 尽量收窄（如限制到某功能目录），免得无关编辑触发慢运行。

#### E.3 — 推荐组合配置

```
1. TDD-first：先写 fixture（上文配方 A/B）
2. RL_SMOKE_CMD 指向那个 fixture       ← 每个开发轮次自动测
3. ralph-lisa auto --engine            ← Ralph 写，Lisa 审
4. §70 cascade 在 CONSENSUS 收尾        ← 收尾时跑完整 PLAN 测试表
5. test-runner 子 agent 做深度排查      ← 把吵闹的失败从主上下文隔离
```

这给出真正的「开发 → 自动测 → 失败上下文喂回」闭环 — 你永远不用手动调框架；
失败会自动在下一轮浮现。

> 注：`RL_SMOKE_CMD` 从 loop 的工作目录运行框架。fixture 用绝对路径。如果 smoke
> 命令 spawn wezterm/Chromium，§171/§173 的 try/finally cleanup + §182 registry
> 会处理 teardown — 不泄漏。

## 何时不用

这个框架用于**文本驱动、顺序、确定性**的测试。它不擅长：

| 场景 | 为什么不行 | 替代 |
|---|---|---|
| 全屏 TUI（vim/htop/ncurses）| 屏幕重绘破坏子串匹配 | `pyte` + `pexpect` |
| 高频游戏式输入 | 50ms 轮询太慢 | 游戏测试框架 |
| 二进制 IO（管道字节）| `get-text` 返回渲染文本 | 直接 subprocess + `stdin/stdout.write()` |
| 视觉回归 / 像素 diff | `screenshot` 只用于诊断 | playwright 快照测试或 Applitools |
| 大规模并行（100+ pane）| 一个 macro = 一次一个 pane | k6 / locust / artillery |
| race 敏感的 flake | 子串匹配捕捉不到时序 | 用确定性 mock |

如果你的测试符合上面多于一项，这个框架就是错的工具。选领域专用的测试框架。

## §197 — AI 输出断言(deterministic floor)

AI 输出非确定性, 但**属性 / schema 断言是确定性的**(§192 design §5)。§197 ships 5 个属性断言原语 + 一个组合器, 全部在 `cli/src/ai-output-assert.ts`:

- `assertValidJson(text)` — JSON 解析检查
- `assertHasFencedCodeBlock(text, { language? })` — 围栏代码块检查;`language` 可选过滤语言标签
- `assertNoSecret(text)` — 5 模式族 (Bearer / api-key / sk- / AKIA / password=), source-of-truth `cli/src/test-logs.ts:17`
- `assertLengthInRange(text, { min?, max? })` — 字符数边界检查
- `assertMatchesSchema(text, schema)` — JSON-Schema 最小子集 (`type / required / properties / items`);未知 keyword 抛 usage error (fail-closed)
- `runAiOutputChecks(text, invariants[])` — 组合器, 跑多个 invariant 返回 `{ results[], allPass }`

**CLI 入口**: `ralph-lisa ai-output-check --content <path> --invariants <path> [--json <out>]` — exit 0 全 pass / 1 任一 fail / 2 usage error。

**invariant JSON 形状** (tagged union):

```json
[
  { "kind": "valid-json" },
  { "kind": "fenced-code", "language": "ts" },
  { "kind": "no-secret" },
  { "kind": "length", "min": 10, "max": 5000 },
  { "kind": "schema", "schema": { "type": "object", "required": ["name"], "properties": { "name": { "type": "string" } } } }
]
```

**何时用**: 任何 AI 生成内容的测试都应该跑 §197 deterministic floor + (§198 deferred) LLM-judge 上层 advisory。§192 §7 lock: LLM 判定**不是**单独 gate, 是 §197 deterministic 之上的 annotated signal。

**何时不用**: 文件 / API / file:line 存在性检查 (需要 codebase 上下文 → §198 LLM-judge territory)。run-N variance 也 deferred。

## §198 — LLM-as-judge harness(artifact-not-gate)

§197 deterministic断言之上,某些主观/语义判断需要 LLM 介入: "this answer covers the user's question?" / "this UI achieves the goal?" / "this screenshot looks correct?"。§198 ships **LLM-as-judge primitive**,§4C / §5 / §6 都消费它。**核心纪律**(§192 §7):**judgement 是 artifact, 不是 gate**;§197 deterministic 才是 gate, LLM verdict 是 floor 之上的 advisory signal。

**核心模块**: `cli/src/llm-judge.ts` ships `runLlmJudge(args)` + `cmdLlmJudge(args)`。

**3 modes**:
- `heuristic` (默认,无需 LLM key): rubric = `{must_contain?: string[], must_not_contain?: string[]}`;score = (matched checks) / (total checks);verdict = score >= threshold ? 'pass' : 'fail'。
- `llm`: 注入 provider (`Provider` type)即可调真模型;provider 返回 `{score, rationale}`;受 `cost_cap_ms` (default 30000) per-call timeout 约束 (Promise.race + clearTimeout/unref,无 timer leak)。
- `off`: stub `{verdict:'unknown', score:0}`,不调 provider 也不写 cache —— 无 LLM key 环境的安全默认。

**Threshold-independent cache** (Lisa R1 narrow lock): cache 落在 `.dual-agent/llm-judge/<sha256(mode||model||tplhash||inputhash)>.json`,只存 `{score, rationale, model_id, prompt_template_hash, input_hash, mode}` —— **不存 verdict**。`verdict` 在每次 read 时从 `score >= threshold` 重算,同一个 provider call 可服务任意 threshold,无 stale verdict 风险。

**Secret-scrub** (复用 §197 SoT): 调用前 input + rubric 全过 `SECRET_PATTERNS` from `cli/src/test-logs.ts:17` —— 防止 secret 进 hash / 进 cache / 进 provider 转录。

**CLI 入口**: `ralph-lisa llm-judge --input <path> --rubric <path> --threshold <n> [--mode <m>] [--json <out>]` —— rc=0 pass / rc=1 fail / rc=2 unknown 或 usage 错误。`--json` 写完整 artifact (含 hash + mode + verdict + rationale)。

**Rubric 形状** (heuristic 模式):

```json
{ "must_contain": ["expected phrase A", "expected phrase B"], "must_not_contain": ["forbidden phrase X"] }
```

也可以直接给字符串 rubric (作为 single substring check);llm 模式下传给 provider 的就是这个对象的渲染版。

**何时用 LLM-judge**:
- 上层 advisory 信号(§192 §7): "这个 AI 回答是否真的解答了用户问题?" / "这个 UI 屏幕看上去对吗?"
- 配合 §197 deterministic floor 一起跑;deterministic 不通过直接 fail,deterministic 通过后 LLM verdict 作为额外 annotated signal。

**何时不用**:
- 单测 / 集成测里测确定性逻辑 —— 用 §197 primitives + 普通 assert。
- 安全断言 —— 用 §197 `assertNoSecret`,不要让 LLM "判断" 是否泄密(deterministic regex 才靠谱)。
- gate 决策 —— LLM 永远是 artifact, 永不当独立 gate。如果 deterministic floor 通过但 LLM verdict='fail',按 §192 §7 默认记录到 audit log 但不阻塞;升级阻塞需要明确 §9 surfaced decision。

**Bounded budgets** (§192 §7): `max_rationale_chars=500` (truncate); `cost_cap_ms=30000` (per-call timeout 通过 Promise.race + clearTimeout 严格上限); `iteration_cap=1` (single-shot; multi-call self-consistency deferred 到 §192-followup)。

## §199 — UI goal-achievement assertion(composite outcome + advisory LLM verdict)

§192 design §6: 上线一个表单不是看 `assert-text "Success"` 就够 —— 真正要确认的是**用户的目标是否达成**: 网络请求真发出去了 / DOM 真切到 Confirmed / 整个 journey 没出错。§199 ships `assert-ui-goal` step type 在 playwright skill 里组合这 3 个 deterministic check, 加上一层 §198 LLM-judge advisory 作为上面的 fuzzy signal。

**纪律** (§192 §7 lock): LLM-judge 永远是 annotation, **不当 gate**。step 通过 / 失败完全由 deterministic floor 决定。LLM verdict 写进 `stepResult.llm_judge` 给人/工具消费, 但绝不影响 `step.pass`。

### Step schema

```json
{
  "type": "assert-ui-goal",
  "goal": "submit and see Thanks",
  "network_expect": { "url_pattern": "/submit" },
  "dom_expect": { "selector": "#status", "text": "Thanks" },
  "negative": true,
  "screenshot_path": "/tmp/result.png",
  "llm_mode": "heuristic",
  "llm_threshold": 0.5,
  "llm_rubric": { "must_contain": ["submit", "Thanks"] }
}
```

**Spec-level intent**: `Spec` 也可以带顶层 `goal?: string` field, 给当前 spec 的所有 assert-ui-goal step 提供 fallback intent。

### Deterministic floor (3 checks; AND semantics)

每个 check 可有可无;present 的全部 pass → step pass;任一 fail → step fail。

- **`network_expect.url_pattern`**: 在 `[network] <url>` 行里做**字面子串匹配**(NOT regex; metachars 字面处理 — Lisa R1 narrow lock)。
- **`dom_expect.selector` + `text?`**: selector 必须 visible (via `runner.isVisible`), 而且(若给了 text)`runner.getText(selector)` 必须 `includes(text)`。
- **`negative: true`**: 扫 `runner.getCapturedEvents()` 里有没有 `[pageerror]` 或 `[console] [error]` — 复用 §195 `assert-no-console-error` 的正则。

### LLM-judge advisory (§198 wiring)

- `runSpec(spec, runner, { llmJudgeRunner })` 通过 `opts.llmJudgeRunner` 注入 §198 `runLlmJudge`。无 LLM key 环境可省略 — step 仍按 deterministic floor 跑。
- step 配置里给 `llm_mode` (`'heuristic' | 'llm' | 'off'`) + `llm_rubric` + 可选 `llm_threshold` → 触发判定;否则不调 judge。
- **Text-only payload** (Lisa R1 narrow): judge 拿到 `{goal, page_text, deterministic_results}` 文本, 不读 screenshot 字节 (vision-LLM 推到 §192-followup)。`page_text` 是 `runner.getText(dom_expect.selector)` 的内容(无 dom_expect 则 omit)。`deterministic_results` 是 `network=X; dom=X; negative=X` 的扁平摘要。
- 结果按 §198 cache identity 缓存 (threshold-independent); `stepResult.llm_judge` 注入完整 JudgeResult (含 verdict / score / rationale / cached / hashes)。
- `screenshot_path` (若给) 仅作为 artifact pointer: `runner.screenshot(path)` 落盘给人工 review, 不喂 judge。

### 何时用 assert-ui-goal

- 端到端验证一个完整用户旅程的**结果**, 而不只是中间状态。
- 一个 step 串联 network + dom + negative + (advisory) LLM 4 个维度, 比铺 4 个独立 step 紧凑且断言强度更明确。

### 何时不用

- 单元 / 集成里测确定性逻辑 — 直接用 §195 `assert-text` / `assert-visible` 等基础原语。
- 不需要跨 layer 综合判断的简单页面 — `assert-text "Thanks"` 够用就别上 §199。
- LLM 不能用作 gate — 任何需要 hard pass/fail 的安全或合规检查仍走 §197 deterministic 原语。

## §202 — 非编码 slice 的 propose/agree 协议

§201 Phase A 把 verifyLisaAttest 接到 checkLisa + 双向 gate 后, §202 Phase B 给非编码 slice 装上**协议层**: Ralph 提议 shape, Lisa 显式同意, 双方 mutual CONSENSUS 时自动落 trusted artifact。替换了原来 4 个静态 escape 词 (`doc-only`/`config-only`/`single-rename`/`process-only`) 的设计 — 现在是动态的 per-slice 协议。

### 6-entry shape whitelist

`cli/src/auto-tdd.ts:57` `ESCAPE_REASON_WHITELIST` 扩到 6 个:
- `doc-only` — 纯文档修改
- `config-only` — 配置 / 数据文件修改, 无代码
- `single-rename` — 单文件 / 单标识符重命名, 无语义变化
- `process-only` — 工作流 / PLAN.md / CLAUDE.md 文档级
- **`analysis-only`** (NEW) — RCA / 排查 / postmortem; 输出是文本结论
- **`research`** (NEW) — 设计 / 研究 slice, 输出是 `docs/<name>-design.md`

### Ralph 怎么提议

Ralph R1 [PLAN] body 加两个字段:

```
**Tests**: none (proposed-shape: analysis-only)
**Shape-Justification**: <≥40 字 + ≥1 个 file:line 或 artifact 引用>
```

`parseRalphProposal(body)` 在 `cli/src/auto-tdd.ts:72` 解析。

### Lisa 怎么同意

Lisa R1 [PASS] body 加两个字段:

```
Agreed-Shape: analysis-only
Shape-Rationale: <≥40 字 + ≥1 个引用; verifyLisaAttest 仍要通过>
```

`parseAgreement(body)` 在 `cli/src/lisa-attest-parser.ts:177` 解析; secret-scrub 后才 hash。

### Artifact 写入

Ralph R1 [CONSENSUS] transition 自动写 `.dual-agent/non-tdd-agreement-<step>.json`, 由 `cmdSubmitRalph` 负责 (per §200 设计 — Lisa 不该 mutate trusted state)。写之前 5 个 precondition:

1. Ralph 的 R1 [PLAN] body (从 work.md 或 history.md) 有 parseRalphProposal 完整字段
2. Lisa 最近的 [PASS] body 有 parseAgreement 完整字段
3. **proposed-shape === agreed-shape** (mismatch → block)
4. `verifyLisaAttest` 通过 (Phase A 跨检 — quality_score=true + unverified.length===0)
5. 文件还不存在 (幂等 — 重复写 → `non-tdd-agreement-already-exists` block)

任一失败 → `non-tdd-agreement-prerequisites-not-met` block, exit code 1。

Artifact 进 §144 trusted-path 白名单 — closeout 时 `Verified: .dual-agent/non-tdd-agreement-<step>.json` 通过。

### First-tag 强制

新 step 的**第一个非-[QUESTION]** Ralph submission **必须**是 `[PLAN]` / `[RESEARCH]` / `[CLARIFY]`。其他 tag → block `r1-first-tag-not-allowed`。

- `[QUESTION]` 允许做开局 (问个问题), 但**不算 round**, 下一个非-QUESTION submission 仍然受 first-tag 强制。
- 现有 step (history.md 已有非-QUESTION Ralph 记录) 不受这个 rule 影响。
- Opt-out: `RL_R1_FIRST_TAG_OFF=1` (audit-named; CI / 测试 fixture 可用)。

### 中途撤回 shape (shape-change-pending)

Lisa 走到中途发现 scope 漂移, 不想要原来的 shape 了:

Lisa submit `[NEEDS_WORK]` (或 `[CLARIFY]`) body 加 field:

```
Shape-Change-Requested: doc-only
```

`cmdSubmitLisa` (NOT checkLisa — 后者 stays pure) 自动写 `.dual-agent/non-tdd-shape-change-pending-<step>.json`。

Ralph 接下来 [CODE]/[FIX] → block `shape-change-pending-not-acked` 指向 `ralph-lisa ack-shape-change` cli。

User 拍板 ack:

```sh
ralph-lisa ack-shape-change --slice <step-slug> --reason "<context>"
```

→ 删 pending file + 写 `.dual-agent/shape-change-audit.jsonl` 审计条; Ralph 可继续。

### Post-CONSENSUS cascade short-circuit

`handleMutualCompletion` 见 artifact 存在且 `shape !== 'code-task'` → 跳过 §70 tier cascade, 直接写 `gate-status.json` `{status: passed, reason: 'non-tdd shape: <X>'}`。非编码 slice 不会因为没有 test runner 输出被 block。

### 何时用 §202 协议

- **真正的非编码任务**: RCA / 文档 / 配置 / 设计 / 单文件改名

## §203 user-behavior-analytics-local (本地行为数据采集 + 聚合)

§203 为 RLL 添加 **本地优先** 的用户行为分析能力 —— 在 §54-§61 token 采集链 / §101 telemetry-shipper / §103 privacy opt-in 基础上, 把 RLL 工作流的细粒度事件聚合成可读 report, 用于产品分析和企业管控审计。

### 4 个新事件类型 (§203 扩展 §101 EventType union 3→7)

`cli/src/event-shipper.ts:EventType` union 从 3 个 (`token-usage` / `submit-event` / `review-event`) 扩到 7 个:

| 事件类型 | 触发点 | 字段 (除通用 `captured_at` / `user_email` 外) |
|----------|--------|----------------------------------------------|
| `model-invocation` | session-jsonl reader (§61 扩展) | `agent` / `model_id` / `role` / `latency_ms` / `prompt_tokens` / `completion_tokens` |
| `narrow-event` | `cmdSubmitLisa` 收到 `[NEEDS_WORK]` | `step` / `lisa_round` / `narrow_rule` (从 §149 backticked kebab-case 解析) |
| `consensus-event` | `handleMutualCompletion` mutual CONSENSUS close | `step` / `total_rounds` / `narrows_count` / `slice_duration_ms` (从 history.md 解析) |
| `command-event` | `cli.ts` 顶层 `process.on('exit')` | `cmd_name` / `duration_ms` / `exit_code` (privacy: **绝不**含 args/env/cwd) |

### 4 个 append-helper API

`cli/src/event-shipper.ts` 导出 (§197 SECRET_PATTERNS scrub 套在 `appendCommandEvent`):

- `appendModelInvocationEvent(stateRoot, e)`
- `appendNarrowEvent(stateRoot, e)`
- `appendConsensusEvent(stateRoot, e)`
- `appendCommandEvent(stateRoot, e)` — `cmd_name` 写入前 scrub Bearer/sk-/api-key/AKIA/password=

### CLI: `ralph-lisa user-behavior-analyze`

本地聚合 + 渲染 report (markdown 或 JSON):

```bash
# 全部默认 — 当前用户 / 全时段 / md 输出 / stdout
ralph-lisa user-behavior-analyze

# 限定用户 + 时段 + JSON + 写文件
ralph-lisa user-behavior-analyze \
  --user xiaomi@example.com \
  --since 2026-05-01T00:00:00Z \
  --until 2026-05-24T00:00:00Z \
  --format json \
  --out /tmp/behavior-report.json

# admin scope (企业管控 — 不过滤 user_email)
ralph-lisa user-behavior-analyze --scope admin --format md
```

报告含 4 个 H2 段:
1. **Activity Timeline** — ASC-sorted 事件列表, 每行一句话摘要
2. **Agent + Model Breakdown** — pivot `model-invocation` + `token-usage` (后者 `model_id='unknown'`, latency=0 — bridge §55-§61 旧数据)
3. **Ralph-Lisa Interactions** — submits/reviews/consensus 计数 + narrow rule breakdown + 平均 narrow/slice
4. **Dev Task Summary** — 每 slice rounds/narrows/duration/outcome (`closed` / `active` / `deadlock`)

### 3 个生产钩子

| 钩子位置 | 默认 | opt-out 环境变量 |
|----------|------|-----------------|
| `cli/src/cli.ts` 顶层 dispatch `process.on('exit')` (command-event) | **on** | `RL_COMMAND_EVENT_OFF=1` |
| `cli/src/commands.ts:cmdSubmitLisa` `[NEEDS_WORK]` 分支 (narrow-event) | on | `RL_NARROW_EVENT_OFF=1` |
| `cli/src/commands.ts:handleMutualCompletion` mutual CONSENSUS close (consensus-event) | on | `RL_CONSENSUS_EVENT_OFF=1` |

`npm test` wrapper (`cli/package.json:test`) 全局设 `RL_COMMAND_EVENT_OFF=1`, 避免测试套件污染 live super-rll/.dual-agent (F12/F13 / T1-T3 snapshot 隔离测试)。

### 隐私契约 (与 §103 telemetry-privacy-opt-in 对齐)

- **本地优先**: §203 默认仅 **本地** 写 `.dual-agent/*.jsonl`。上传 (走 §101 event-shipper → `/v1/telemetry/cli`) 需用户显式 `~/.rll/telemetry.json:opted_in=true` (§103 default-deny)。
- **command-event 字段白名单**: 仅 `cmd_name` / `duration_ms` / `exit_code` / `user_email` / `captured_at`。`cmd_name` 在 append 前 §197 SECRET_PATTERNS scrub。**绝不**含命令行参数、环境变量、工作目录、shell 历史。
- **narrow-event 字段白名单**: 仅 `step` / `lisa_round` / `narrow_rule` (kebab-case identifier)。**不**保存 NeedsWork-Rationale 正文。
- **consensus-event 字段白名单**: 仅 `step` / `total_rounds` / `narrows_count` / `slice_duration_ms`。**不**保存提交内容或 review 内容。

### 事件源 (analyzer 读哪些文件)

`loadLocalEvents` 读 `.dual-agent/` 下:
- `token-usage.jsonl` (§55-§58)
- `submit-events.jsonl` / `review-events.jsonl` (§101)
- `model-invocations.jsonl` / `narrow-events.jsonl` / `consensus-events.jsonl` / `command-events.jsonl` (§203 NEW)

`captured_at` 之外 / `sinceMs..nowMs` 之外的事件由 `loadLocalEvents` 在加载时过滤; 未来时间戳的事件会被丢弃 (避免 fixture 时间 bug — 见 §203 R2 narrow #1)。

### 与 §101 `telemetry-push` 的关系

`ralph-lisa telemetry-push --dry-run` 现在能 batch 全部 7 个事件类型 (§203 R2 dogfood 验证)。真打上传仍受 §103 consent gate 保护。
