# RLL 一看就懂 — 它是什么 + 它怎么干活

> RLL = **R**alph–**L**isa **L**oop。
> 一句话:**两个 AI 结对开发——Ralph 写代码、Lisa 审代码,轮流来,谁也绕不过谁,直到双方都点头才算完。**
> 目的:让 AI 写出来的代码**真能用、真被验证过**,而不是"看起来过了"。

---

## 一、它是什么(30 秒看懂)

```
        ┌─────────────────────────────────────────────┐
        │                  RLL 一个 session             │
        │                                               │
        │   ┌──────────┐   轮流交接   ┌──────────┐       │
        │   │  Ralph   │ ───────────▶ │   Lisa   │       │
        │   │ 开发 AI   │ ◀─────────── │ 审查 AI   │       │
        │   │ 写/改/测  │   (turn)     │ 挑刺/把关 │       │
        │   └──────────┘              └──────────┘       │
        │        ▲                                       │
        └────────┼───────────────────────────────────────┘
                 │ 任务从 3 个口进来
     ┌───────────┼────────────┐
   你直接说    喵吉/飞书      GitHub issue
   (终端)      (军师转派)     (自动抓取)
```

- **Ralph**(开发)= 写计划、写测试、写代码、跑测试、根据反馈改。
- **Lisa**(审查)= 独立复核,挑出真问题,只有真过了才放行。
- **turn(回合)**= 任何时刻只有一方能动手。Ralph 提交 → 轮到 Lisa 审 → Lisa 出结论 → 轮回 Ralph。**绝不能自己审自己、自己放自己过。**

**为什么要两个 AI?** 一个 AI 容易"自我感觉良好"——写完说"应该没问题"。RLL 强制一个独立的审查者用证据复核:**测试真跑了吗?断言真能抓到 bug 吗?还是只是恒真的空断言?** 这就堵住了"假装完成"。

## 二、它怎么干活(开发工作流程)

一个开发任务叫一个 **sub-slice(子切片)**。每个 slice 按固定回合走,**像闯关,每关 Lisa 把门**:

```
  复杂任务                          简单任务
  ─────────                         ─────────
  R0 [CLARIFY]  先跟你把范围问清楚      (跳过)
       │        (哪些做/哪些不做)
       ▼
  R1 [PLAN]     架构/方案讨论          R1 [PLAN]
       │
       ▼
  [TDD-PLAN]    把测试用例锁死         (跳过)
       │        (进开发前先定好怎么验)
       ▼
  [CODE] 只写测试   先写"会失败"的测试    [CODE] 直接实现
       │        (证明测试真能抓问题)
       ▼
  [CODE] 写实现   把测试跑绿
       │
       ▼
  [FIX] × N     按 Lisa 挑的问题改
       │
       ▼
  [CONSENSUS]   双方都点头 → 关单 ✅
```

### 每个回合在干嘛(给人话)

| 回合标签 | 大白话 | Lisa 在把什么关 |
|---|---|---|
| `[CLARIFY]` | "动手前先问清楚:这事到底要做啥、不做啥?" | 防止做歪、做大、过度设计 |
| `[PLAN]` | "我打算这么搭架子,你看方向对不对" | 方向/设计是否合理 |
| `[TDD-PLAN]` | "进开发前,把每条测试用例(命令+预期+判定)先定死" | 测试计划是否覆盖真实场景 |
| `[CODE]` 只写测试 | "先写测试,而且它现在**应该是红的**(没实现当然失败)" | 测试是不是真能抓 bug(不是恒真空壳) |
| `[CODE]` 写实现 | "现在写代码,把刚才的红测试跑绿" | 实现质量 + 测试真绿 |
| `[FIX]` | "Lisa 挑出问题了,我改,并说明她为啥对" | 问题真被修了 |
| `[CONSENSUS]` | "双方都认可,收工" | 互相确认,不是单方面拍板 |

> **核心纪律:先写会失败的测试,再写实现。** 一条测试只有"先证明它在没实现时会红、有实现后变绿",才算真测试。否则就是"恒真断言"——永远过,等于没测。RLL 把这条做成了硬门禁。

## 三、什么挡住"假装完成"(门禁 / discipline)

RLL 不靠"相信 AI 说的话",靠**机械证据**:

- **反 vacuous(反空壳)**:断言必须先红后绿;改坏被测物能让它失败,才算数。
- **测试结果核对**:Ralph 报的"X/Y passed"会跟真实执行日志比对,**对不上直接挡**——不能编测试结果。
- **双向 attest(签字)**:Ralph 提交要附"我跑了什么命令、什么结果";Lisa 放行要附"我复核了哪几行、哪个证据"。**Lisa 不能rubber stamp,Ralph 不能假报。**
- **复杂度门禁**:任务多复杂由结构性判据机械判定(改几个文件?跨不跨系统?接不接外部?),不由 AI"感觉"。复杂的强制走全套礼仪。
- **关单级联(§70 cascade)**:双方 CONSENSUS 后,自动把这个 slice 的测试用例**再真跑一遍**,绿了才真关单;红了打回重做。

> 一句话:**"看起来过了"不算过,"有证据证明真过了"才算过。**

## 四、任务从哪来(3 个进料口)

```
   ① 你直接说          ② 喵吉/飞书           ③ GitHub issue
   (在终端打字/语音)    (军师把活派过来)       (仓库里开 issue)
        │                   │                    │
        ▼                   ▼                    ▼
   ────────────────── 进 RLL 的任务总线 ───────────────────
                            │
                            ▼
                   开成一个 sub-slice,Ralph/Lisa 开干
```

1. **你直接说** — 在终端跟 Ralph 对话,"/task 做个 X" 就开一个新 slice。
2. **喵吉/飞书(军师)** — 你在飞书发一句话,军师(喵吉)理解后通过"总线"派给对的 RLL session,干完它把结论聚合回报给你。你只面对一个对话面,不用盯 N 个终端。
3. **GitHub issue** — 仓库里开一个带 ` ```json {"kind":"task",...}``` ` 块的 issue,RLL 的 **poller(守护进程)** 定时轮询、自动抓进来开成 slice,干完在 issue 上贴 `[task-reply]` 评论 + 关闭 issue。**完全无人值守的闭环。**

## 五、一个真实例子(本 session 真发生的)

> 这套机制不是 PPT,下面是它**真跑过**的一次闭环:

1. 有人在 GitHub `super-rll` 仓开了 **issue #5**:"Test-Harness 升级需求(3 类)",带 `kind:task` 块。
2. RLL 的 **poller** 定时轮询,自动发现 #5 → 解析 → 注入成一个 sub-slice → 启动 Ralph/Lisa 干活。
3. 任务太大(3 类 23 条),Ralph 没硬塞,**拆成 A/B/C 三步**,跟用户确认从 A 开始。
4. **slice A**(确认流程门禁)走完整流程:`CLARIFY`(锁范围)→ `TDD-PLAN`(锁 7 条测试)→ 只写测试(红)→ 写实现(绿)→ Lisa 挑出 2 个真问题(测试覆盖不够/空壳)→ `FIX` → 双方 `CONSENSUS` 关单。
5. 期间 Lisa 真抓到 bug(比如"反空壳只覆盖了 1/4 的检查"),Ralph 改完用证据说明"她为啥对"。
6. 全程 `cli 2663/2663` 测试绿。slice B、C 待续,做完会自动在 #5 上回报 + 关单。

**看到了吗:** 一句"开个 issue" → 自动进来 → 被严格开发 + 审查 → 结果自动回报。中间没有"假装写完"的空间。

## 六、术语小抄(看到这些词不慌)

| 词 | 人话 |
|---|---|
| **slice / sub-slice** | 一个独立的开发任务单元 |
| **turn(回合)** | 当前轮到谁动手(Ralph 或 Lisa) |
| **NEEDS_WORK** | Lisa 说"还不行,这几处要改" |
| **PASS** | Lisa 说"这一轮可以" |
| **CONSENSUS** | 双方都点头,slice 关单 |
| **CHALLENGE** | Ralph 不同意 Lisa,拿证据反驳(不是盲从) |
| **gate(门禁)** | 自动卡点,不满足条件就挡住提交 |
| **anti-vacuous(反空壳)** | 测试必须真能抓 bug,不能恒真 |
| **喵吉 / 军师(junshi)** | 唯一对人的飞书入口 + 任务路由调度者 |
| **poller** | 定时轮询 GitHub issue 的守护进程 |
| **bus(总线)** | 任务/回报在 session 之间传递的文件信箱 |

## 七、一句话总结

**RLL 让"AI 写代码"变成"AI 结对开发 + 强制审查 + 证据门禁"——**
任务从你、从飞书、从 GitHub 进来,被一个开发 AI 和一个审查 AI 一轮轮打磨,
每一步都要拿真证据过关,谁也不能假装完成,直到双方都认可才收工,结果自动回报给你。
